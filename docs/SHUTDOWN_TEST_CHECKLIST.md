# PC 에이전트 종료/상태보고 통합 테스트 체크리스트

배포 전 회귀 테스트용. 각 항목은 `pc_agent.log`(설치 경로의 로그 파일)와 접점제어기/MQ 쪽 로그를 같이 대조해서 확인한다.

## 0. 사전 준비

- [ ] 테스트 PC에 최신 `pc_agent.exe`, `pc_agent_service.exe`, `installer.exe` 배포 확인 (버전 `1.0.3` 로그/설치 화면에 표시되는지)
- [ ] 서비스 재시작 완료, `pc_agent.log`에 `starting UDP listener` 새로 찍힌 것 확인 (구버전 잔존 여부 체크)
- [ ] 접점제어기 IP, PC UDP 포트 확인

## 1. 정상 시스템 종료 → OFF 보고

| 절차 | 확인할 로그 | 통과 기준 |
|---|---|---|
| 시작 메뉴 → 시스템 종료 클릭 | `sent status=shutdown_accepted reason=windows_shutdown_detected` 또는 `windows_service_shutdown` | 종료 클릭 후 수 초 내 로그 발생 |
| 같은 순간 | `payload=b'#PC_STATUS:0&'`가 감지 즉시 3번 연속(약 150ms 간격) | 버스트 3회 확인 |
| 이후 12초간 | `reason=periodic_shutdown` 3초 간격으로 반복 | 12초 지나면 push 중단 (하단 §3 참고) |
| 접점제어기/MQ 쪽 | PC 상태가 OFF로 전환되는지 | 접점제어기가 실제로 OFF 인지하는지 |

## 2. 접점제어기 OFF 명령 수신 반응 속도

| 절차 | 확인할 로그 | 통과 기준 |
|---|---|---|
| 접점제어기에서 OFF(코드 1001) 명령 전송 | `received command code=1001 from <접점제어기IP>` | 명령 도착 후 **매우 짧은 시간(수십 ms) 내** 로그 발생 |
| 바로 다음 줄 | `sent response code=2002 status=shutdown_accepted` | 수신 직후(같은 타임스탬프 근처) 응답 |
| 접점제어기가 명령 보낸 시각과 PC 로그 시각 비교 | — | 30초 이상 지연되면 원인이 PC가 아니라 접점제어기/서버 쪽 execute→전송 지연인지 먼저 확인 (과거 사례: ESP32 쪽에서 40초 지연 발견됨) |

## 3. push 윈도우 종료 후에도 조회 응답 정상 동작

| 절차 | 확인할 로그 | 통과 기준 |
|---|---|---|
| 종료 감지 후 12초 초과 대기 | 이후 `sent status=` 자동 push 로그가 더 안 찍힘 | push 중단 확인 |
| 이 상태에서 접점제어기가 STATUS 조회(코드 1002) 전송 | `sent response code=2002 status=shutdown_accepted` | push은 멈췄어도 조회엔 여전히 OFF로 정확히 응답 |

## 4. 종료 중단(취소)/절전 후 자동 복구

| 절차 | 확인할 로그 | 통과 기준 |
|---|---|---|
| 종료 트리거 후 실제 종료를 인위적으로 중단(예: 업데이트 취소, 강제로 화면 복귀) | `starting UDP listener`가 **새로 찍히지 않는지** 확인 (동일 프로세스 지속 여부) | 새 프로세스 시작 로그 없으면 "중단된 세션 지속" 케이스 |
| `shutdown_accepted` 진입 후 120초 경과 대기 | `WARNING ... shutdown appears to have been aborted; resetting shutdown tracking` | 정확히 약 120초 뒤 발생 |
| 그 직후 | 다음 정기 주기(20초)에 맞춰 `sent status=online reason=periodic` | 복구 즉시 online을 강제 push하지 않고 정상 주기까지 대기하는지 확인 |

## 5. 완전 종료 후 재부팅 → 정상 online 복귀

| 절차 | 확인할 로그 | 통과 기준 |
|---|---|---|
| 실제로 전원 LED/팬까지 완전히 꺼지는 것 육안 확인 후 재부팅 | `starting UDP listener`가 **새로** 찍힘 | 새 프로세스 시작 확인 (진짜 재부팅 증거) |
| 재부팅 후 | `sent status=online reason=periodic` | shutdown 관련 잔여 상태 없이 정상 online만 보고 |

## 6. 멀티 네트워크 인터페이스(Wi-Fi + 이더넷 동시 연결)

| 절차 | 확인할 로그 | 통과 기준 |
|---|---|---|
| Wi-Fi, 이더넷 둘 다 연결한 상태로 서비스 구동 | `sent status=... to <Wi-Fi 대역 broadcast>:포트`, `sent status=... to <이더넷 대역 broadcast>:포트` 각각 찍히는지 | 두 대역 모두에 각각 directed broadcast로 전송되는지 |
| 접점제어기가 실제 연결된 대역(예: Wi-Fi) 쪽 | 접점제어기가 정상 수신 | 다른 대역이 함께 나가도 실제 통신에 지장 없는지 |
| ipconfig로 각 인터페이스 서브넷마스크 확인 | — | `/24`가 아닌 마스크(`/16`, `/23` 등)가 섞여 있어도 각각 정확한 broadcast로 계산되는지 |

## 7. 설치 시 브로드캐스트 주소 수동 입력

| 절차 | 확인할 로그/화면 | 통과 기준 |
|---|---|---|
| `installer.exe` 대화형 실행, "브로드캐스트 주소를 직접 입력하시겠습니까?(Y/N)"에서 **N** | 자동 계산 방식 그대로 동작 | §6과 동일하게 자동 계산 |
| 동일 질문에서 **Y** + 유효 IP 입력 | `config.json`의 `startup_notify_host`가 입력한 값 그대로 저장 | `pc_agent.log`의 `sent status=... to <입력한 IP>`로 고정 전송 (자동 계산 로직 미적용) |
| **Y** + 잘못된 형식(`abc` 등) 입력 | 설치 시 `install error: 올바른 IP 주소 형식이 아닙니다` | 에러로 즉시 중단, 잘못된 값으로 저장되지 않음 |

## 8. 평상시(ON) 주기 보고

| 절차 | 확인할 로그 | 통과 기준 |
|---|---|---|
| 정상 부팅 상태로 방치 | `sent status=online reason=periodic` 간격 측정 | 기본값 20초 간격 (설치 시 `--status-interval`로 변경했다면 그 값) |

## 9. 이벤트로그 폴링 스로틀 (성능 회귀 확인)

| 절차 | 확인할 방법 | 통과 기준 |
|---|---|---|
| 접점제어기와 ACK를 빈번히 주고받는 상태에서 명령 수신 테스트 | §2와 동일하게 명령 수신 지연 측정 | 지연 없이 즉시 반응 (과거 wevtutil 병목 재발 안 함) |
