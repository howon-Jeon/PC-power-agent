# PC 원격 전원 제어 시스템 기획서 v3

## 1. 배경 및 목적

### 배경
- 다원DNS 스크린골프 무인매장 솔루션(카카오 협업) 개발 중 PC ON/OFF 원격 제어 기능 필요
- 기존 C# MQTT 방식에서 구조적 한계 및 배포 문제로 방향 전환

### 목적
- **단기**: 카카오 스크린골프 무인매장 PC 원격 ON/OFF 제어
- **중장기**: 키오스크, 타사 매장 PC 등 다양한 환경에 적용 가능한 범용 제품으로 확장
- 접점제어기 1대 = PC 1대 제어 구조로 다수 매장, 다수 디바이스 운영

### 핵심 방향
- PC에 복잡한 클라우드 클라이언트 없이 가벼운 에이전트 하나만 설치하면 동작
- 사용자가 짧은 설치 코드 하나만 입력하면 자동으로 설정 완료

---

## 2. 기존 방식에서 변경한 이유

### C# → Python으로 언어 변경
- C#으로 빌드한 .exe 배포 시 Windows 코드 서명 인증서 문제 발생
- Python으로 개발 후 PyInstaller로 단일 .exe 파일로 빌드하여 배포

### MQTT 직접 구독 → UDP 수신 방식으로 변경
- 범용 제품화 시 타사 PC에 다원 MQTT 인프라 클라이언트를 설치하게 만드는 것은 영업/설치 장벽
- 접점제어기(ESP32)가 이미 다원 서버와 MQTT로 연결되어 있으므로 해당 연결 재사용
- PC 쪽은 UDP 리스너만 있는 경량 에이전트로 충분

## 3. 전체 시스템 구성

### 디바이스별 역할

| 디바이스 | 역할 |
|---|---|
| 다원 서버 (AWS) | MQTT 브로커, 접점제어기에 명령 전달, 매장별 AES 키 및 포트번호 발급/관리, 설치 코드 발급 |
| 접점제어기 (ESP32) | PC ON: 릴레이 접점으로 전원버튼 시뮬레이션 / PC OFF: UDP 브로드캐스트 송신 / PC 상태: MQTT로 다원 서버에 전달 |
| PC 에이전트 (Python .exe) | UDP 리스너 백그라운드 상시 동작, 명령 수신 및 실행, PC 상태 UDP로 접점제어기에 응답 |

### 디바이스 구성 예시

```
매장 A
  접점제어기1 ↔ PC1  (포트: 47001, 키: 매장A-1 전용 키)
  접점제어기2 ↔ PC2  (포트: 47002, 키: 매장A-2 전용 키)

매장 B
  접점제어기1 ↔ PC1  (포트: 47001, 키: 매장B-1 전용 키)
  접점제어기2 ↔ PC2  (포트: 47002, 키: 매장B-2 전용 키)
```

접점제어기 1대가 PC 1대를 전담 제어. 매장/디바이스별로 독립된 키를 사용하여 타 매장 디바이스가 다른 매장 PC를 제어하는 것을 원천 차단.

### 통신 흐름

**PC OFF 명령 흐름**
```
다원 서버
 → (MQTT) 접점제어기
 → (UDP 브로드캐스트, 지정 포트) 해당 포트를 열고 있는 PC만 수신
 → AES-GCM 복호화 및 검증
 → 종료 실행
 → (UDP 유니캐스트) 접점제어기에 상태 응답
 → (MQTT) 다원 서버에 상태 전달
```

**PC 상태 확인 흐름**
```
다원 서버
 → (MQTT) 접점제어기
 → (UDP 브로드캐스트, 지정 포트) 상태 요청 명령
 → PC가 UDP 유니캐스트로 상태 응답 ("온라인")
 → (MQTT) 다원 서버에 상태 전달
```

**PC ON 흐름** (기존 방식 유지)
```
다원 서버
 → (MQTT) 접점제어기
 → 릴레이(접점) ON → 전원버튼 신호 인가 → PC 부팅
```

## 4. 핵심 설계 결정 사항

### 4-1. PC 식별 방식: 포트 번호

**방식**
- 사용자가 PC 에이전트 설치 시 번호를 직접 입력 (예: 1, 2, 3...)
- 입력한 번호에 따라 포트가 자동 계산됨 (번호 1 → 포트 47001)
- 접점제어기가 해당 포트로 브로드캐스트 전송
- 그 포트를 열고 있는 PC만 자동으로 수신

**포트 범위**
- 47001 ~ 47099
- 해당 대역은 일반 프로그램과 충돌 가능성 낮음
- 99개로 매장 내 PC 수 기준 충분

**이 방식의 장점**
- IP 주소 관리 불필요 (DHCP로 IP 바뀌어도 영향 없음)
- 별도 PC_ID 매칭 로직 불필요 (포트가 곧 식별자)
- 구조가 단순하고 설치가 간편함

### 4-2. 통신 방식: UDP 브로드캐스트

- 접점제어기가 255.255.255.255로 브로드캐스트 전송
- 포트 번호로 PC를 구분하므로 IP를 알 필요 없음
- PC → 접점제어기 응답 시에는 수신 패킷의 출발지 IP로 유니캐스트 전송

**네트워크 전제 조건**
- 접점제어기와 PC가 반드시 같은 서브넷에 있어야 함
- 매장 네트워크가 VLAN으로 분리되어 있으면 안 됨 (브로드캐스트가 VLAN을 넘지 못함)

### 4-3. 암호화: AES-GCM 256비트

- 모든 UDP 패킷을 AES-GCM 방식으로 암호화
- 키 길이: 256비트 (32바이트)
- AES-GCM은 암호화와 무결성 검증을 동시에 제공 (변조 감지 가능)
- 매장별, 디바이스별로 독립된 키 사용

### 4-4. 암호화 키 관리: 설치 코드 방식 (방법 B)

**키 배포 흐름**
```
1. 다원 앱/웹에서 매장별 6자리 설치 코드 발급
2. 설치 기사가 PC 에이전트 설치 시 설치 코드 입력
3. PC 에이전트가 설치 코드로 다원 서버에 1회 접속
   → AES 키 + 포트 번호 수신
4. 수신한 키와 포트를 로컬에 암호화 저장
5. 이후 다원 서버와 직접 통신 없이 UDP로만 동작
```

**이 방식의 장점**
- 사용자는 짧은 코드(6자리)만 입력하면 됨
- 최초 1회만 서버 접속 후 이후 서버 의존성 없음
- 매장/디바이스별 독립 키로 보안성 확보
- 키 노출 시 해당 매장/디바이스 키만 교체하면 됨

**키 저장**
- PC 에이전트 로컬에 암호화된 형태로 저장
- 설치 코드는 1회용으로 사용 후 만료 처리 (서버에서 관리)

### 4-5. PC 상태 전송: UDP (MQTT 미사용)

- PC 에이전트는 다원 서버에 직접 MQTT 연결하지 않음
- PC 상태(온라인/오프라인)는 UDP로 접점제어기에 전송
- 접점제어기가 MQTT로 다원 서버에 대신 전달
- 단말기 IP는 수신 패킷의 출발지 IP를 그대로 활용 (별도 설정 불필요)

## 5. PC 에이전트 동작 방식

### 최초 설치 시
```
1. .exe 설치 (더블클릭 → 다음다음완료)
2. 설치 화면에서 설치 코드 입력 (6자리, 다원 앱/웹에서 발급)
3. 다원 서버에 1회 접속 → AES 키 + 포트 번호 수신 및 로컬 저장
4. 방화벽 포트 자동 오픈 (배치파일 자동 실행)
5. Windows 서비스 자동 등록 및 시작
```

### 부팅 후 상시 동작
```
1. PC 부팅 시 Windows 서비스로 자동 실행
2. 지정된 포트(예: 47001) UDP 오픈 및 수신 대기
3. 브로드캐스트 패킷 수신 시:
   → AES-GCM 복호화
   → 복호화 실패 시 무시 (다른 키로 암호화된 패킷)
   → 명령 코드에 따라 동작 실행
4. 명령 실행 후 접점제어기로 상태 응답 UDP 전송
```

### 보안 처리
- AES-GCM 복호화 실패 시 자동 무시 (타 매장/디바이스 패킷 차단)
- GCM 특성상 패킷 변조 시 자동 감지 및 무시
- 설치 코드는 1회용, 사용 후 서버에서 만료 처리

## 6. UDP 페이로드 프로토콜

### PC 식별 방식
- 페이로드 안에 PC_ID 등 별도 식별자 없음
- **포트 번호 자체가 PC 식별자** 역할
- 접점제어기가 지정 포트로 브로드캐스트 → 해당 포트를 열고 있는 PC만 수신
- 페이로드는 순수하게 명령 코드만 포함

### 패킷 구조
- 전체 페이로드는 AES-GCM 256비트로 암호화하여 전송
- 복호화 실패 시 자동 무시 (타 매장/디바이스 패킷 차단)

### 명령 코드
> ⚠️ 상세 명령 코드는 프로토콜 문서 수신 후 확정 예정

| 방향 | 내용 |
|---|---|
| 접점제어기 → PC | 명령 코드 (예: 1001 = OFF) |
| PC → 접점제어기 | 상태 응답 코드 (예: 온라인/오프라인) |

### 확정 후 업데이트 필요 항목
- 전체 명령 코드 목록
- PC → 접점제어기 응답 코드 체계
- 패킷 내 필드 순서 및 구분자

## 7. 방화벽 자동 설정 (배치파일)

설치 시 배치파일이 자동 실행되어 방화벽 인바운드 규칙을 추가함. 사용자가 수동으로 방화벽 설정할 필요 없음.

**처리 내용**
- 설치 시 입력한 번호에 해당하는 포트만 인바운드 허용
- 기존에 동일 규칙이 있으면 삭제 후 재등록
- 서비스 제거 시 방화벽 규칙도 함께 삭제

## 10. 테스트 절차

### 1단계 — PC 에이전트 단독 테스트 (ESP32 없이)
- 같은 LAN의 다른 PC에서 테스트용 UDP 송신기로 패킷 전송
- AES-GCM 암호화 포함한 패킷 수신 및 복호화 정상 동작 확인
- 종료 실행 코드는 주석 처리 상태로 먼저 테스트

### 2단계 — 실제 종료 테스트
- 종료 실행 코드 활성화 후 실제 shutdown 테스트

### 3단계 — Windows 서비스 테스트
- 서비스 설치 후 PC 재부팅, 자동 실행 확인
- 로그인 없이 백그라운드 동작 확인

### 4단계 — ESP32 통합 테스트
- 펌웨어 업데이트 후 다원 서버 → ESP32 → PC 전체 경로 통합 테스트
